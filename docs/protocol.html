Stream Exchange Protocol

SEP v0.1.0 — An open protocol defining how stream providers and stream consumers communicate in a decentralized content ecosystem.

View JSON Schemas Roadmap

Protocol Overview

SEP is content-agnostic and meta-first. Streams exchange enriched metadata, not full media. Content is lazy-loaded by the consumer only when needed.

Consumer —[intent]→ Provider —[enriched payload]→ Consumer —[telemetry]→ Provider

Five Core Schemas

1. Enrichment Envelope

The foundational data structure. Every content item carries source metadata, semantic enrichment scores, emotional tone analysis, and session-fit indicators.

2. Consumer Intent

How consumers express what they want: session type, duration, category weights, hard filters, context, and disclosure level.

3. Provider Response

How providers return results: enriched payloads with confidence, suggested arcs, errors, and redirects.

4. Provider Manifest

How providers advertise capabilities: exchange models, content types, guardrails, rate limits.

5. Telemetry

Optional engagement signals: per-item viewing data and session-level satisfaction. Always opt-in.

Exchange Models

SEP supports three exchange patterns. Both sides declare support during the initial handshake.

Model A: Curated Payload

Simplest model. Consumer sends intent, provider returns a curated set. Single request-response. Provider has more influence over results.

Consumer --[intent]--> Provider
Consumer <--[curated payload]-- Provider
Recommended

Model C: Negotiated Exchange

Multi-round exchange with progressive refinement. State tokens enable continuity. Consumer can reveal more over successive rounds.

Round 1:
  Consumer --[intent]--> Provider
  Consumer <--[payload + token]--> Provider
Round 2:
  Consumer --[intent + token]--> Provider
  Consumer <--[refined payload]--> Provider

Model B: Full Index Browse

Provider exposes entire metadata index. Maximum consumer sovereignty. Most expensive for provider.

Consumer --[query]--> Provider Index
Consumer <--[raw results]-- Provider
Consumer applies local PAE filtering

Enrichment Envelope

The core data structure of SEP. Every content item in a stream carries an enrichment envelope.

Top-Level Fields

FieldTypeStatusDescription
sep_version string required Protocol version this envelope conforms to
item_id uuid required Globally unique identifier for this content item
source object required Origin platform, URL, content type, and duration
meta object required Title, creator, publication date, tags, language, thumbnail
enrichment object required LLM-generated semantic scores and classifications

skill_transfer_score0–1Actionable skill/knowledge transferred

session_fitobjectBooleans: good_opener, good_builder, good_peak, good_closer

Emotional Tone Flags

Boolean flags for harmful content patterns. Enable consumers to set hard exclusion filters:

FlagDescription
rage_baitDesigned to provoke anger for engagement
humiliationDerives entertainment from humiliating individuals
shock_contentRelies on shock value to drive engagement
inflammatoryDesigned to inflame divisions or provoke heated reactions
sexually_explicitContains sexually explicit material
violenceContains graphic violence or glorifies violent behavior

Consumer Intent

Describes what a consumer is requesting from a provider. The consumer always controls disclosure.

FieldTypeStatusDescription
sep_version string required Protocol version
consumer_id string required Consumer identifier (can be anonymous)
intent.session_type enum required composed (structured arc), browse (open exploration), search (targeted query)
intent.target_duration_minutes integer required Desired session length in minutes
intent.weights object required Category weight distribution (e.g., fitness: 0.25, humor: 0.20)
intent.filters object required Hard exclusion filters for harmful content, energy bounds, language
disclosure_level enum required minimal, standard, or full
telemetry_opt_in boolean required Whether to share engagement telemetry
intent.context object optional Time of day, session count, state token for multi-round exchanges

Disclosure Levels

LevelWhat the provider sees
minimalWeights and filters only. No behavioral history.
standardWeights, filters, plus recent anonymized session summaries.
fullWeights, filters, and full behavioral profile. Provider can hyper-personalize.

      "time_of_day": "morning",

Provider Response

The provider's response to a consumer intent. Supports payloads, errors, and redirects.

FieldTypeStatusDescription
sep_version string required Protocol version
provider_id string required Provider identifier
response_type enum required payload, error, or redirect
payload.items array conditional Array of enrichment envelopes, ordered by relevance
payload.confidence 0–1 conditional Overall confidence that this payload satisfies the intent
payload.suggested_arc array optional Suggested session arc for optimal pacing
payload.state_token string optional Opaque token for stateful multi-round exchanges
capabilities object optional Stateful support, full index browse, telemetry exchange, max payload size

Provider Manifest

A provider's self-declaration document, published at a well-known endpoint for discovery and negotiation.

FieldTypeStatusDescription
provider_id string required Unique provider identifier
provider_name string required Human-readable provider name
endpoint uri required Base URL of the SEP API endpoint
supported_models array required Exchange models: negotiated, curated_payload, full_index_browse
supported_content_types array required Content types: video, podcast, music, article
guardrails object optional Published guardrail policy: URL, version, transparency flag
enrichment_schema_version string required Enrichment schema version used by this provider
max_payload_size integer required Maximum enrichment envelopes per response
rate_limit object optional Requests per minute and daily cap

Privacy Architecture

State lives primarily on the consumer side. The provider sees only what the consumer chooses to share.

Disclosure Levels

Consumers control what providers see. Three granularity levels from anonymous weights to full behavioral profiles.

Opaque State Tokens

Providers store session continuity tokens, not behavioral profiles. State is functional, not surveillance.

Voluntary Telemetry

Engagement data is aggregated before sending. Always opt-in. Creates a transparent value exchange.

Multi-Provider Queries

Consumers can query multiple providers with different disclosure levels. No provider sees the full picture.

Versioning

SEP uses dual versioning: semantic versioning for the protocol (0.1.0, 0.2.0) and date-based snapshots for schema releases (spec/2026-02-28/). Enrichment schemas are versioned independently from the protocol.
